440 /*=============================================================================
441 *
442 * The following wtbarr typedef is used in the fits_read_wcstab() routine,
443 * which is intended for use with the WCSLIB library written by Mark
444 * Calabretta, http://www.atnf.csiro.au/~mcalabre/index.html
445 *
446 * In order to maintain WCSLIB and CFITSIO as independent libraries it
447 * was not permissible for any CFITSIO library code to include WCSLIB
448 * header files, or vice versa. However, the CFITSIO function
449 * fits_read_wcstab() accepts an array of structs defined by wcs.h within
450 * WCSLIB. The problem then was to define this struct within fitsio.h
451 * without including wcs.h, especially noting that wcs.h will often (but
452 * not always) be included together with fitsio.h in an applications
453 * program that uses fits_read_wcstab().
454 *
455 * Of the various possibilities, the solution adopted was for WCSLIB to
456 * define "struct wtbarr" while fitsio.h defines "typedef wtbarr", a
457 * untagged struct with identical members. This allows both wcs.h and
458 * fitsio.h to define a wtbarr data type without conflict by virtue of
459 * the fact that structure tags and typedef names share different
460 * namespaces in C. Therefore, declarations within WCSLIB look like
461 *
462 * struct wtbarr *w;
463 *
464 * while within CFITSIO they are simply
465 *
466 * wtbarr *w;
467 *
468 * but as suggested by the commonality of the names, these are really the
469 * same aggregate data type. However, in passing a (struct wtbarr *) to
470 * fits_read_wcstab() a cast to (wtbarr *) is formally required.
471 *===========================================================================*/
